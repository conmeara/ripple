From AI event
to rendered pixel

Every message, tool call, and approval in Ripple's chat passes through 5 transformation stages before the user sees it. This document maps every stage, every filter, and every customization point — so you can change exactly what's shown.

Stage 1
Event Emission
service.ts
Provider emits typed events. Persisted to DB + broadcast on EventEmitter.
Stage 2
Projection
agent-runtime-ui-projection.ts
Events mapped to UI chunks. Filtering happens here — dropped events never reach the renderer.
Stage 3
Part Assembly
ai SDK · applyChunkToAssistantParts()
Chunks are accumulated into typed message parts. Tool calls accumulate input then output.
Stage 4
Part Grouping
assistant-message-item.tsx
Parts are grouped (exploring, task), ACP types normalized, and tool steps collapsed when a final reply exists.
Stage 5
Component Render
renderPart() · per-component files
Each part type dispatches to a dedicated React component. Some part types render null.
Stage 1

Event Emission

The AI provider emits 12 typed events through AgentProviderEventSink.emit(). Each is persisted to the database and broadcast on a Node EventEmitter, then streamed to the renderer via TRPC subscription.

assistant_text_delta
payload: { delta: string }
Streaming text fragment — emitted many times per response turn
assistant_message
payload: { text: string }
Complete text message — used when provider doesn't stream deltas
reasoning
payload: { delta?: string, text?: string }
Extended thinking content — Claude's internal reasoning before replying
tool_start
payload: { toolName, tool, providerId, input }
AI is about to call a tool — begins a tool call accumulation
tool_update
payload: { toolName, input?, output? }
Streaming progress of a running tool call
tool_end
payload: { toolName, output, error? }
Tool call completed — contains final result or error
file_change
payload: { changes: [{ path, kind, diff }] }
AI proposed a file edit — shown as a text label in the chat (not a full diff view)
approval_request
payload: { kind, status, reason, command, ... }
AI needs user permission to proceed — renders interactive approve/deny card
status
payload: { label?, capabilities?, sessionInit? }
Run lifecycle update — heavily filtered; most status events are dropped
activity
payload: { kind: thinking|editing|… }
Always dropped by projector — activity indicators come from provider state, not this event
usage
payload: { inputTokens, outputTokens, totalCostUsd, … }
Token usage and cost — shown in session info panel, not in chat
error
payload: { message: string }
Run failed — shown as an error chunk in the message stream
TRPC Bridge — Events flow from main process → renderer via trpcClient.agentRuntime.chat.subscribe() as an observable. The renderer receives { type: "event", event } messages and passes each through the projector. De-duplication by event ID prevents double-rendering.
Stage 2

Projection — Events → Chunks

AgentRuntimeUIProjector.project(event) is the primary filter. It maps each raw event to zero or more UI chunks. Events that return [] are completely invisible in the UI.

Event type → Chunk type(s) Visible? Notes
assistant_text_delta → text-delta shown Streams text into the text part. If reasoning was open, closes it first.
assistant_message → text-delta shown Only used if no text_delta events were seen first. Non-streaming fallback.
reasoning → reasoning-start + reasoning-delta shown Renders in collapsible thinking block. Closed automatically when text starts.
tool_start → tool-input-start shown Tool name normalized (commandExecution→Bash, fileChange→Edit/Write, webSearch→WebSearch, imageView→Read, MCP→mcp__server__tool).
tool_update → tool-input-available or tool-output-available shown Streaming tool progress. If payload has output, emits output chunk; otherwise input chunk.
tool_end → tool-output-available or tool-output-error shown Final tool result. If error field present → output-error; otherwise output-available.
file_change → data-agent-runtime {kind:"file_change"} shown Renders as a plain text label "Updated proposal diff" — not an inline diff viewer.
approval_request → data-agent-runtime {kind:"approval"} shown Always rendered. Creates an interactive approve/deny card.
status → data-agent-runtime {kind:"status"} or [] filtered Only passes if: has a label AND not capabilities/sessionInit AND not "Loaded Codex/Claude context". Many status events are silently dropped.
activity → [] (always empty) dropped Never reaches the UI. Activity indicators (thinking, editing…) come from the run status/state, not this event.
usage → message-metadata shown Populates token counts, cache info, cost, duration — shown in session/usage panel.
error → error shown Emits error chunk that terminates the stream and shows the error message.

Tool name normalization — Codex and Claude use different tool names. The projector normalizes them: commandExecution → Bash, fileChange → Edit or Write, webSearch → WebSearch, imageView → Read, MCP tools become mcp__serverName__toolName. This happens in toolNameFromPayload().
Stage 3

Part Assembly

The AI SDK accumulates chunks into typed message parts — the stable objects that React components actually render. Text parts accumulate string content; tool parts accumulate input then output as separate lifecycle states.

text part
{ type: "text", text: string, state: "streaming"|"done" }
All text-delta chunks are concatenated into this single part's text field. State goes streaming → done.
reasoning part
{ type: "reasoning", text: string, state: "streaming"|"done" }
Extended thinking content. Closed when text starts streaming. Normalized to tool-Thinking type for rendering.
tool-{ToolName} part
{ type: "tool-Bash"|"tool-Read"|…, toolCallId, toolName, state, input, output, providerExecuted }
State lifecycle: input-streaming → input-available → output-available (or output-error). Input and output accumulated separately.
data-agent-runtime part
{ type: "data-agent-runtime", id, data: { kind, label, payload } }
Wraps approval requests, file_change events, and status labels. kind controls which sub-component renders.
message-metadata
{ type: "message-metadata", messageMetadata: { inputTokens, outputTokens, totalCostUsd, durationMs, … } }
Not rendered in the chat stream — consumed by AgentMessageUsage component for session info display.
error part
{ type: "error", errorText: string }
Terminates the stream. Rendered as an error message in the chat.
Stage 4

Part Grouping & ACP Normalization

Before rendering, parts are transformed in AssistantMessageItem: Codex ACP types are normalized, consecutive exploring/task tools are grouped, and tool steps are collapsed when a final text reply exists.

ACP/Codex normalization
Codex sends tools as "tool-Read README.md" (verb + filename in type). normalizeAcpParts() maps these to canonical Claude Code types.
Read→Read Run→Bash List→Glob Search→Grep Thought→Thinking Fetch→WebFetch Edit→Edit Write→Write
After normalization, both Claude and Codex use identical part types — same render path.
Exploring group
3 or more consecutive read-only tool calls are grouped into a single collapsible row — prevents the chat from becoming a wall of file reads.
tool-Read tool-Grep tool-Glob tool-WebSearch tool-WebFetch
Renders as AgentExploringGroup — a compact collapsed row. Nested tools (inside a Task) are excluded from grouping.
Task tool grouping
1 or more consecutive task management tool calls are grouped. Any task management call triggers grouping.
tool-TaskCreate tool-TaskUpdate tool-TaskGet tool-TaskList
Renders as AgentTaskToolsGroup. Nested tools (colon in toolCallId) render inside AgentTaskTool instead.
Tool step collapsing

When a message has both tool calls and a final text reply after them, the tool steps before the final text are collapsed into a CollapsibleSteps component — showing "N steps" with an expand/collapse toggle.

Collapsed (hidden by default)
tool calls [0 … n-1]
↳ shown inside CollapsibleSteps
Visible always
text reply [last text part]
↳ rendered directly below

Plan file operations (Write/Edit to plan files) inside collapsed steps show a mini "Created plan / Updated plan" indicator rather than the full card. The last plan operation's full card appears after the CollapsibleSteps.

Where to change what's displayed

Every display decision is localized to a specific function or set. Here's exactly where to make each type of change — no guesswork required.

Hide code snippets in AI text replies
src/renderer/components/chat-markdown-renderer.tsx
The MemoizedTextPart renders text through a custom markdown renderer. Code blocks and inline code have their own renderer functions you can replace or remove.
// In chat-markdown-renderer.tsx:
components: {
  code: ({ children }) =>
    <span className="font-mono">{children}</span>,
  pre: ({ children }) =>
    <div className="...">{children}</div>,
}
Hide tool call output (Bash stdout etc.)
src/renderer/features/agents/ui/agent-bash-tool.tsx
Each tool type has its own component. To hide Bash output, edit AgentBashTool to not render part.output. To hide a tool entirely, return null from renderPart().
// In renderPart() — assistant-message-item.tsx:
if (part.type === "tool-Bash") {
  return null // hide entirely
  // or render without output:
  return <AgentBashTool
    part={{...part, output: null}}
  />
}
