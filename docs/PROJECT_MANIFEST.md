# Project Manifest: Ripple

## Overview

Ripple is a local-first desktop app for creating short motion graphics through plain-English prompts and frame-anchored comments instead of manual After Effects workflows. It reuses the strongest parts of the existing 1Code desktop/chat/agent foundation, then reshapes the product around HyperFrames so motion designers, editors, marketers, founders, and agency teams can create title cards, lower thirds, social promos, and explainers without learning GitHub, coding tools, dependency installs, or project setup.

## Tech Stack

| Layer    | Technology | Notes |
|----------|-----------|-------|
| Frontend | Electron + React 19 + TypeScript + Vite | Existing desktop renderer foundation; Ripple-specific shell embeds HyperFrames-native editing and preview surfaces. |
| Styling  | Tailwind CSS + Radix UI + Motion + Ripple design wrappers | Keep current component system, but replace upstream 1Code/21st styling and product chrome end-to-end. |
| State    | Jotai + Zustand + TanStack React Query | Preserve existing local UI, chat, and async state patterns where they fit Ripple’s project/comment/revision workflow. |
| Routing  | App-shell state and pane navigation (no dedicated URL router today) | Project-first onboarding, composition switching, and `Chat` / `Comment` pane modes should be handled inside the desktop shell. |
| Backend  | Electron main process + tRPC IPC + HyperFrames CLI orchestration on Node.js 22+ | Main process owns project creation, environment checks, preview/render commands, export, and hidden revision isolation. |
| Database | SQLite + Drizzle ORM + better-sqlite3 | Persist local app state such as projects, sessions, comments, revisions, providers, and settings; HyperFrames project files live on disk under `~/Ripple`. |
| Testing  | TypeScript checks today; add focused integration/smoke coverage for project creation, preview sync, comments, and export | Automated testing is currently light in the upstream repo and needs Ripple-specific coverage. |
| Linting  | `tsgo --noEmit` currently; repo-wide lint/format standard to be finalized during rebrand | Existing code contains legacy ESLint/Biome directives, but Ripple should consolidate on one enforceable standard. |

## Domain Model

- **Project**
  Fields: `id`, `name`, `slug`, `localPath`, `aspectRatioPreset`, `activeCompositionId`, `templateId`, `setupStatus`, `createdAt`, `updatedAt`
  Purpose: the top-level Ripple workspace stored by default at `~/Ripple/<project-name>`.

- **Composition**
  Fields: `id`, `projectId`, `name`, `filePath`, `dataCompositionId`, `width`, `height`, `parentCompositionId`, `kind`, `createdAt`, `updatedAt`
  Purpose: a HyperFrames HTML composition; `index.html` is usually the entry composition and child compositions live under `compositions/`.

- **Clip**
  Fields: `id`, `compositionId`, `selector`, `type`, `trackIndex`, `startFrame`, `durationFrames`, `assetId`, `styleSnapshot`, `timelineLabel`
  Purpose: a timed visual or media element in a HyperFrames composition using `class="clip"` and timing data attributes.

- **Asset**
  Fields: `id`, `projectId`, `name`, `kind`, `relativePath`, `mimeType`, `width`, `height`, `durationMs`, `importedAt`
  Purpose: imported media such as images, logos, video, or audio stored under `assets/`.

- **CommentThread**
  Fields: `id`, `projectId`, `compositionId`, `anchorType`, `startFrame`, `endFrame`, `elementSelector`, `screenshotPath`, `status`, `createdBy`, `createdAt`
  Purpose: frame/time-based review feedback attached to a scene, range, or visible element context in a motion-designer-friendly workflow.

- **Revision**
  Fields: `id`, `commentThreadId`, `projectId`, `isolatedContextPath`, `prompt`, `provider`, `status`, `previewOutputPath`, `diffSummary`, `createdAt`, `resolvedAt`
  Purpose: one isolated proposal generated by an agent in response to a chat request or comment thread.

- **ExportJob**
  Fields: `id`, `projectId`, `compositionId`, `format`, `qualityPreset`, `outputPath`, `status`, `errorMessage`, `createdAt`, `completedAt`
  Purpose: a HyperFrames-native render/export request for `MP4`, `MOV`, or `WebM`.

- **AgentSession**
  Fields: `id`, `projectId`, `revisionId`, `mode`, `provider`, `conversationSummary`, `createdAt`, `updatedAt`
  Purpose: the preserved 1Code chat/agent foundation adapted to Ripple’s `Chat` and `Comment` workflows.

Relationships:
- A `Project` has many `Composition`, `Asset`, `CommentThread`, `Revision`, `ExportJob`, and `AgentSession` records.
- A `Composition` can contain many `Clip` records and can nest child `Composition` records through `data-composition-src`.
- A `Clip` may reference one `Asset`; an `Asset` can be reused across many `Clip` records.
- A `CommentThread` targets one `Composition` and optionally a specific `Clip` or element/time range.
- A `Revision` belongs to one `CommentThread` or chat request, produces an isolated reviewable change set, and can update one or more `Composition` or `Asset` files.
- An `ExportJob` renders one `Composition` within one `Project` and records the final output artifact.

## Conventions

(default — update when scaffolded)

- File naming: `PascalCase.tsx` for renderer components, `kebab-case.ts` for features/stores/utilities, and `kebab-case.html` for HyperFrames compositions and templates.
- Test files: `*.test.ts` and `*.test.tsx` near source or under `tests/`; add smoke tests for project creation, preview sync, comment revision, and export flows.
- API routes: no repo-first HTTP API is required for local MVP; prefer typed Electron main↔renderer tRPC procedures and wrapper services around HyperFrames CLI commands.
- Commits: conventional commits such as `feat:`, `fix:`, `refactor:`, and `chore:` with Ripple-facing scopes when useful.
- Branches: feature branches per work item; hidden git/worktree or snapshot mechanics are allowed internally for revisions, but user-facing UI must use revision/comment language instead.

## Constraints

- HyperFrames is the source of truth for composition authoring, preview, and render; Ripple must not replace it with React/Remotion scene abstractions.
- Local use must not require account creation, provider selection, or mandatory auth before the user can enter the product shell.
- End users must not need to understand Git, worktrees, dependency installation, or manual project-folder setup.
- The normal create-project flow must create `~/Ripple/<project-name>` automatically, including folder naming, setup, dependency install, and hidden local revision initialization if needed.
- Local rendering depends on Node.js `22+` and FFmpeg; Ripple must detect readiness and explain problems in simple app language instead of raw CLI-only troubleshooting.
- Agent execution must stay inside the active project boundary or an isolated revision context derived from that project.
- Shipped Ripple paths must remove `21st.dev`, `1Code`, upstream deep links, upstream update URLs, mandatory upstream auth, and other hosted-product assumptions from the primary UX.
- MVP excludes cloud sync, multi-user collaboration, mobile clients, and full After Effects export compatibility.

## Success Criteria

### Per-Feature Success

- [ ] User can create or open a Ripple project and enter the main shell without signing in, connecting a repo, or manually choosing a provider first.
- [ ] User can ask for a motion-graphics change in chat or from a frame/time-based comment and see a refreshed HyperFrames preview quickly.
- [ ] User can review a proposed revision with captured screenshot + frame/range context and explicitly accept or reject it.
- [ ] User can export a usable 1080p video in one session, with `MP4` as the default and `MOV` / `WebM` available when needed.

### Factory-Level Success

- [ ] Ripple ships as a fully rebranded product with no primary-path `21st.dev`, `1Code`, or Remotion assumptions.
- [ ] HyperFrames-native editing, preview, template insertion, and local rendering behave reliably on supported Node.js `22+` + FFmpeg environments.
- [ ] Preserved foundation systems that still matter to Ripple (multi-provider agents, automations/inbox, kanban, file viewer, plugins, MCP, and voice) remain available unless intentionally deferred.

## Review Standards

### Spec Compliance

- Enforce project-first onboarding and no mandatory auth gate before the user can create or open a Ripple project.
- Ensure features preserve HyperFrames-native HTML composition rules, timeline semantics, and export behavior instead of introducing React/Remotion-style scene abstractions.
- Require comment revisions to carry captured frame/range context, reviewable output, and explicit accept/reject flows.

### Style

- Reuse the existing Electron/React/Tailwind shell patterns where helpful, but replace all upstream branding, product strings, and hosted-service assumptions with Ripple-specific chrome and language.
- Prefer official `@hyperframes/studio` and `@hyperframes/player` behavior/primitives over bespoke parallel editors when the official tooling already solves the problem.
- Use motion-design-first user language such as project, scene, composition, clip, timeline, comment, revision, preview, and export; avoid repo/branch/worktree jargon in primary UX.

### Security

- Keep agent file access inside the active project boundary or a hidden isolated revision context derived from that project.
- Do not require or assume upstream hosted auth for local use; retain encrypted credential storage only for optional provider/account flows.
- Validate file-system paths, render destinations, and local dependency checks before launching preview or export commands.

### Severity Scale

- **Low**: cosmetic issues, minor inconsistencies
- **Medium**: functional gaps, missing edge cases
- **High**: data loss, security vulnerability, spec violation
