Ripple · {{VERSION}} · what it is and how it works

One perception cache, eighteen instruments,
four opinions on the craft.

Everything in Ripple hangs off one spine: perceive → decide-with-user → assemble → verify → deliver. The CLI is the product and teaches itself through stdout; the skills are the optional craft opinion on top; taste lives in project files, not prompts. Click any command or skill below to open it.

All four scenarios (raw-clips → cut, demo video, interview + b-roll, fiction) run this same spine — the per-scenario differences are almost entirely opinions, which is why the playbooks are thin and the CLI is shared.

The strategy difference in one line: ffmpeg is hands, HyperFrames is imagination, the NLE is the finishing suite — Ripple is the eyes, ears, and editorial judgment loop. Everyone above owns one slice; Ripple owns the loop that connects them and routes to the rest.

Measured on ripple’s own eval harness (reproducible via evals/)
  • Same agent, same task, tooling ablated: Sonnet with raw ffmpeg: 559s / $1.12. With the ripple CLI alone: 158–249s / $0.76–1.11 across two release runs — 2–3.5× faster, and it passes the deterministic checks (endings kept, tails tight, HDR preserved) that the bare run barely scrapes.
  • All four scenarios pass live evals (v0.11): flagship tighten, demo UI-settle from motion curves, interview search in 7.4 minutes, fiction J-cut, multicam sync (24s / $0.21), animation routing.
  • The opinions route correctly (v0.12): service evals pass 10/10 checks — VO → ElevenLabs with a standing voice ID, music → generated to the manifest's duration, b-roll → recut/stock before generation.
  • Triggering: 19/20 against ~20 competing installed video skills — plain-language editing asks route into ripple 3/3 without ever naming it; near-misses stay out.
  • Channel economics: for text-domain questions the index is 4× cheaper than the sheet; behavior questions (the look-down) are decided in the image channel — each question has a home channel.
Honesty box · what is NOT measured
  • No head-to-head benchmark against these tools exists — ours or anyone's. Published video-editing benchmarks target generative editing; nobody scores editorial quality (open ground the prior-art doc flags as ours to take).
  • The fairest proxy we have is the ablation above: rung A ≈ "an agent with no editing tool," which is the world most of these tools also try to improve on — but that is a proxy, not a comparison run against video-use or OpenMontage themselves.
  • Borrowable deterministic metrics for a future benchmark: cut-point-vs-word-boundary distance, boundary embedding distance, audio RMS at cuts, SyncNet offline.
