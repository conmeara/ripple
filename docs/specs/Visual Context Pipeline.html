Ripple Spec / Visual Context Pipeline / May 2026

Fast visual context is a pipeline, not one benchmark.

Ripple makes two separate decisions for every visual: first, how to capture the right pixels; second, how to hand those pixels to the agent without wasting an extra turn. The winning path is the fastest correct capture route plus provider-native image return.

Comments: automatic first message visuals Chats: on-demand visual tools Codex: dynamic tool image result Claude: SDK MCP image result

Mental Model

Decision split
Capture time and handoff time answer different questions.

Warm Engine versus cold CLI is a capture-stage decision. Native image return versus path-only return is a runtime-handoff decision. They should be measured separately, then composed into an end-to-end local pipeline estimate.

Chosen architecture
  • CaptureUse live preview for the current frame. Use warm Engine for timestamped frames and sheets.
  • StoreWrite a scoped local artifact and manifest so the result is auditable and reusable.
  • InjectReturn provider-native image content directly from the tool result.

Pipeline

The same service powers comments and chats. The difference is when the visual is requested: comments request it before the agent run starts; chats request it later when the agent asks to see something.

Stage 01
Intent

A surface asks for a typed visual. The surface does not choose a backend.

Point commentFrame at the comment timestamp.
Range commentFrame sheet for the selected range.
Chat tool callSnapshot or frame sheet on demand.
Stage 02
Capture Route

The Visual Context Service picks the fastest correct source for the visual intent.

Current frameLive preview capture, because those pixels already exist.
Specific timestampWarm Engine render, because it is source-canonical.
Frame sheetWarm Engine samples plus sheet assembly.
Stage 03
Artifact

Ripple writes a safe project-scoped image artifact and keeps structured metadata.

PNGSnapshot or composed sheet.
ManifestTimes, frames, dimensions, and route metadata.
GuardrailsNo symlink escape or arbitrary absolute path trust.
Stage 04
Runtime Handoff

The provider adapter converts the artifact into the provider's native image result.

CodexDynamic visual tool returns inputImage.
ClaudeSDK MCP visual tool returns an image block.
FallbackCLI can still return a path and JSON manifest.
Key rule

The agent should not need a second turn to inspect an image it just asked Ripple to capture. The path and manifest remain useful for auditability, but the image itself is injected in the tool result.

Optimization Goal

Make visual context feel instant by driving app-owned latency below strict budgets, while tracking provider/model wait separately. A slow model choosing when to call a tool is different from Ripple taking too long after the tool has started.

North star
No normal visual-context path should make the user wait on Ripple for more than two seconds.
<2s p95

Current frames should be effectively immediate. Timestamped frames and compact sheets should come from warm, source-valid capture paths. Cold CLI behavior remains a fallback, not the agent path.

Measurement rule
Every row reports app-owned latency, budget status, and the dominant bottleneck.

For automatic comment visuals, app-owned latency is submit-to-visual-ready because that gates first-message image injection. For tool paths, app-owned latency is tool execution after the model calls Ripple. Model wait remains visible, but it is not mixed into the capture budget.

Implementation bias

Optimize the path the agent actually uses. Keep one warm visual-context engine per active project/composition, cache current frames and source-valid sampled frames, precompute range sheets during selection, inject provider-native images directly, and make the eval fail the app-owned budget before it blames provider/model latency. The active preview surface heartbeat is the prewarm signal: it keeps the current-frame identity fresh and warms the source-canonical render path before timestamp snapshots or sheets are requested.

Latest optimization slice

The frame-sheet problem moved from app capture to model tool choice. Range comments precompute a sheet while the user composes, point comments prewarm the current frame before send, comment agents reuse attached visuals when the requested frame or range matches, native tools keep backend choice app-owned, and app-managed timestamp or frame-sheet captures default to the fast browser path with bounded sheet-cell dimensions. In the corrected live matrix, every app-owned visual path is inside budget: automatic comment visuals land in 71-93ms and native tool captures stay at 0-456ms.

Provider Injection

Codex and Claude use different extension APIs, but Ripple gives the agent the same product behavior: the tool result includes the image.

Codex
dynamic tool

Ripple registers dynamic visual tools on the Codex App Server session. When Codex calls ripple.snapshot or ripple.frame_sheet, Ripple captures the visual and returns inputImage content with a data URL.

RegisterbuildRippleVisualDynamicToolSpecs()
Handleitem/tool/call for the ripple namespace.
ReturninputText summary plus inputImage.
Claude
SDK MCP tool

Claude Agent SDK exposes app-defined tools through an SDK-created MCP server. This is not a shell lookup loop. It is Claude's custom-tool lane, and the handler returns a native image content block directly.

RegistercreateSdkMcpServer("ripple_visual_context")
Handleripple_snapshot and ripple_frame_sheet.
Returntext summary plus image base64 content.
Shared core

Both providers call the same visual context core. The only provider-specific part is the final wrapper: Codex wants inputImage, while Claude wants an SDK MCP result with type: "image".
