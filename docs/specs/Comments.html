Spec · Comments

Comments

Comments are Ripple's Frame.io-style review pane. The user is the creative director — they watch the motion piece and drop notes at exact moments. The agents are the editors. Each note becomes an isolated proposed version the user can preview, refine, and accept into Main.

Audience: Motion designers reviewing agent-edited projects
Surface: Right-side review pane
Status: Building
Test markers T-A3 existing test T-A2 test to write Hover any marker for details · jump to Test plan

The workflow

Comments fit into a specific moment in the user's day. Most projects begin in Chats — the user asks the agent to scaffold the comp, build out the motion language, set the timing. By the time they open Comments, they have something playable. Comments is for the round of small, surgical fixes that come next.

01 — Chat
Build the comp
Scaffold the project. Set the motion language. Get to a playable draft.
02 — Comments
Drop the round
Watch the piece. Pause. Type. Hit enter. Move on. Drop ten in a row.
03 — Review
Review & accept
Click each card to see the change. Reply if needed. Accept what works.
Note
Comments ≠ Chats. Chats is for the broad creative direction — open-ended conversation that may rewrite large parts of the project. Comments is for the targeted round of feedback after the comp exists. See Chats for the broader chat surface.

Key concepts

Five terms describe everything Comments does.

Main primary
The accepted project. What the user sees by default in Preview. Only changes when a proposal is accepted.
Comment CommentThread
An anchored review note tied to a frame, range, or element. Holds messages, attachments, and the proposed version.
Anchor frame · range · element
Where the comment lives. Captured automatically from whatever the user is looking at when they start typing.
Proposed version Revision
The agent-generated change set tied to a comment. Lives in isolation until accepted. Full lifecycle in Revisions.
Reply CommentMessage
A follow-up message in the thread. Continues the same agent conversation — does not start over.
Visual context frame attachment
The frame or range image automatically attached to a comment so the agent can see what the user saw. See Visual Context.
02 · Experience

Instant submission

A user dropping a round of notes should never wait. Type, enter, gone. Then on to the next frame, type the next note, enter, gone. The agent runs in the background. The user's flow is never interrupted.

03 · Behavior

Accepting changes

Accept means: bring this proposed version into Main. Everything about how Accept works is in service of one promise — Main cannot be corrupted.

changes ready → user accepts → accepted into Main → other comments auto-pull

Rules

  • Only the accepted comment's change lands in Main. Other in-flight proposals are not touched.T-I5After accept, only this revision applies; thread → acceptedintegration · existing · comment-revisions.test.ts
  • Accept is serialized. A project-level lock prevents two comments from mutating Main at once.T-I3Accepts serialized via project mutexintegration · existing · comment-revisions.test.ts
  • Accept is enabled only when the proposal is ready and based on the current Main. If Main has moved, Accept stays disabled until the proposal absorbs the latest Main — which Ripple does automatically. See the auto-merge cascade below.T-I1Accept enabled only when status=proposedintegration · existing · comment-revisions.test.tsT-I2Stale-base gate: refuses if baseCommit ≠ Main HEADintegration · existing · comment-revisions.test.ts
  • Failure is honest. If accept fails, the card does not pretend it succeeded. Main remains safe and the user gets a clear message.T-I4Failed accept atomically rolls back Mainintegration · existing · comment-revisions.test.ts

Full safety model — patch strategy, base-commit gate, atomic rollback — lives in Revisions.

Cascade rules

  • Stale revisions detected on accept. Every other proposed revision whose baseProjectCommit ≠ new Main is marked updating.T-J1On accept, stale proposed revisions → updatingintegration · existing · comment-revisions.test.ts
  • Cheap replay first. Ripple tries to replay each proposal's patch onto the new Main. This takes milliseconds and zero agent tokens.T-J2Patch replay attempted before agent invocationintegration · existing · comment-revisions.test.tsT-J3Successful replay → status returns to proposedintegration · existing · comment-revisions.test.ts
  • Agent on conflict, silently. If a patch conflicts, the agent is automatically assigned to resolve. The agent does not post a reply — it fixes the conflict and the card returns to Changes ready as if nothing happened.T-J4Failed replay → agent invoked for resolutionintegration · existing · comment-revisions.test.tsT-J5Silent agent fix does NOT post conversation replyintegration · new
  • Accept morphs into a waiting indicator. While a comment is updating, the same agent response stays on the card and the Accept check is replaced by a progress circle. The user knows it'll be ready soon and can keep moving.
  • Other accepts are not blocked. Comments that are still Changes ready remain acceptable. Only the ones whose base actually moved are updating.T-J6Other ready accepts remain enabled during cascadeintegration · new
  • Conflicts stay agent-owned. If replay conflicts, the card stays in the updating shape while the agent resolves it and then returns to Changes ready.T-J7Conflict resolution stays updating until readyintegration · existing · comment-revisions.test.ts
Note
The agent's silent conflict-resolution run uses the same isolation model as a normal revision. It does not appear in the conversation history because it is not a creative response — it is plumbing. The user only sees the result.

Edge cases

Agent responded without project changes

Useful for questions, clarifications, or QA notes. The card lands on No changes needed with an emerald dot and the agent response still visible. Accept is disabled. The user can reply to push for a change, or delete the comment.

Agent needs more context

The card shows the agent's response in the thread. Accept is hidden because there are no project changes to accept. The user replies with more context, opens the conversation in Chat, or deletes the comment.

Agent failed to start or errored

The card moves to Refresh needed. The user can click the bottom-right Refresh action to restart the same agent run, or delete the comment if it is no longer useful.T-D4Refresh needed: amber dot, bottom-right Refresh actioncomponent · new

Ripple crashed mid-run

On startup, any revision left in preparing or running is reset to queued and re-enters the queue.T-K1preparing/running on startup reset to queuedintegration · existing · comment-revisions.test.ts Anything that lost its workspace is recovered by the agent before it can be accepted again.T-K2Lost workspace requeues agent recoveryintegration · existing · comment-revisions.test.ts See Failure Recovery.

How to read the plan

  • Each behavioral claim in the spec carries an inline test marker. Hover the marker to see the test name, type, and status.
  • Existing tests are in green. They pin the behavior today — modifying that behavior should fail the test first (the Prove-It pattern from the TDD skill).
  • New tests are in blue. These are gaps — write the test before the change, watch it fail, then make it pass.
  • Test names read as specifications. Each name describes the behavior under test in plain English. If a test name doesn't make sense as a sentence about user-facing behavior, rewrite it.
