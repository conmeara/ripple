01 / Product Shape

The User Installs Ripple; Agents Get Motion Know-How Automatically

Ripple should feel like a motion-design app, not a prompt-bundle installer. The user connects Codex or Claude Code, opens a local HyperFrames project, and asks for edits in normal creative language. Ripple quietly gives the provider enough context to find official HyperFrames guidance, use bundled project tools, inspect the current composition, and produce media assets such as transparent subject cutouts and overlays.

Bundled by the app

Ripple packages the official HyperFrames skills from the upstream repository and stages them into provider-native skill surfaces at run time.

Visible to agents

Agents see skill metadata and Ripple policy before acting. They load full skill bodies only when the request matches the skill.

Invisible to most users

The product talks about projects, compositions, preview, comments, revisions, exports, media assets, and overlays. Skill roots stay advanced/debug language.

Ideal shape Ripple behavior is prompt policy; HyperFrames expertise stays searchable skills
Always-on Ripple prompt Visual verification, native tool priority, revision boundaries, app-managed commands, project safety.
Official HyperFrames skills Authoring, media prep, transparent overlays, animation libraries, registry blocks, preview, render, export.
Project context AGENTS.md or CLAUDE.md, active composition, current frame/range, comments, assets, conversation history.
Agent work Edits the motion project, creates assets, verifies visuals, then reports in product language.

Media workflow emphasis

The policy should nudge agents toward `hyperframes-media` when a user asks for a keyed subject, a transparent cutout, a foreground person/object layer, or a reusable overlay asset. The user-facing idea is an editable motion asset with transparency, not a CLI exercise.

  • Put generated media in a project asset location, normally under `assets/` or a clearly named project media folder.
  • Reference the output from the composition as an overlay layer.
  • Verify transparency visually, then inspect or render when a final export matters.
06 / Skill Roots, Plugins, And Conflicts

Ripple Adds App-Managed Roots Without Owning The User's Global Setup

A user may already have HyperFrames skills installed globally. That is acceptable. Ripple's job is to make the app-bundled HyperFrames version available and preferred inside Ripple projects while preserving provider-native discovery and user-installed skills.

07 / QA Contract

The Test Contract Proves Prompt Policy, Skill Visibility, And Project Context

The architecture should be treated as correct only when provider discovery, prompt injection, project context, and live invocation all work. These are the target checks that should stay green after the refactor.

Codex discovery App Server sees all 15 Ripple-bundled HyperFrames skills after project-native staging, with no first-party Ripple skill defaulted.
Claude discovery SDK initialization lists HyperFrames skills, including `hyperframes-media`, while Ripple visual behavior arrives through `systemPrompt.append`.
Live skill invocation Codex and Claude can both load `hyperframes-media` for the transparent-overlay media workflow.
Regression checks Focused context tests, agent runtime tests, workflow coverage, e2e visual-context evals, package smoke hooks, and `git diff --check` guard the contract.
