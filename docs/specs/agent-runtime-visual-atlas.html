A map for debugging, explaining, and changing the runtime

One product runtime, two provider-native execution engines.

Ripple takes a chat or comment, resolves the current project/workspace, assembles live HyperFrames context, attaches visual evidence when appropriate, and then hands the run to Codex or Claude through their native APIs. The provider streams back normalized events; Ripple persists those events, updates run state, and projects the result into the app transcript.

Ripple is the source of truth Threads, runs, statuses, approvals, transcript projections, and revision lifecycle live in SQLite.
Providers stay native Codex gets app-server turns and dynamic tools. Claude gets SDK query, MCP servers, skills, and permission hooks.
Visual context is first class Comment stills/sheets can attach automatically. Snapshot and frame-sheet tools remain available during the run.

02 · Scenario lens

The same runtime changes shape by entry point.

Use these scenarios when debugging why an agent saw a frame, edited a worktree, asked for permission, or reused provider continuity.

Normal chat

Text-first run with on-demand visual tools

A regular chat starts from the current project or chat workspace. It does not receive an automatic image; the model should call native visual tools when visual judgment matters.

03 · Provider split

Codex and Claude share a contract, not an implementation.

Codex App Server

codex_app_server
process
Spawns bundled Codex CLI with app-server args and speaks JSON-RPC over stdio.
threading
Ripple keeps a clean product transcript. Codex starts an app-server thread for the turn and saves provider ids when emitted.
context
Uses developerInstructions for Ripple policy, optional fallback project notes, local MCP snapshot, Codex skills, prompt mentions, and prepared attachments.
visuals
Dynamic tools ripple.snapshot and ripple.frame_sheet return image content items. Approved visual shell commands can be auto-allowed inside the workspace.
approval
JSON-RPC approval requests become app approvals for commands, file changes, permissions, network, and user-input questions.

Claude Agent SDK

claude_agent_sdk
process
Imports @anthropic-ai/claude-agent-sdk and runs sdk.query with the bundled Claude Code executable.
threading
Uses SDK session ids when available via resume, while Ripple still owns the app transcript and run state.
context
Loads Claude runtime capabilities, system prompt, settings, skills, plugins, agent mentions, attachments, and the merged prompt bundle.
visuals
Adds an SDK MCP server named ripple_visual_context with ripple_snapshot and ripple_frame_sheet tools.
approval
SDK canUseTool and elicitation hooks call the shared approval bridge, returning allow, deny, or edited input.
04 · Context stack

What the model receives before it starts acting.

The runtime prompt is not just the user text. It is a layered product snapshot, shaped by workspace, comments, visuals, attachments, provider policy, and project notes.

User promptThe direct request, plus parsed mentions for agents, skills, tools, and prepared uploaded files.
Chat historyVisible conversation context from Ripple's transcript, not arbitrary provider-side history.
Live contextProject, composition, editing target, preview source, preview time/frame, comment anchor, revision, and export source.
Comment visualsOptional automatic still frame or frame sheet with a compact prompt description, only when accepted by attachment limits.
Provider policyRipple visual tool-choice policy, app behavior policy, HyperFrames guidance, and provider-specific instruction channels.
Notes + skillsAGENTS.md or CLAUDE.md, app-managed HyperFrames skills, visual-context skill, and project skill roots.
Tool envWorkspace-rooted environment with bundled ripple and hyperframes, visual endpoint tokens, and bridge directories.

The mental model

Ripple composes the prompt as a product-aware workspace snapshot. Provider adapters then translate that snapshot into native primitives: Codex app-server turn input or Claude SDK prompt/options.

Why notes are not state

AGENTS.md and CLAUDE.md are durable team notes. Runtime-specific facts like current frame, preview source, revision id, and comment anchors are injected into the run prompt instead.

Workspace boundary

The resolved workspace path becomes the provider cwd and writable root. Approval logic checks commands and file paths against that boundary before allowing risky work.

05 · Visual context lane

The runtime has three ways to show the model what the user sees.

Visual evidence is either attached before the provider starts or exposed through native tools during the run. The policy asks the model to use these first for visual judgment.

Current frame snapshot

Runtime context can resolve a preview time or frame into a VisualCurrentFrameSnapshot. Native tools can then capture at=current or an exact timestamp.

Comment frame sheet

Range comments can attach sampled motion as a sheet before execution starts, with prompt text explaining cell timing and range semantics.

Runtime visual tools

Codex receives dynamic tools; Claude receives an SDK MCP server. Both routes call the same visual command runner and return image content to the model.

06 · State machine and event stream

Provider noise becomes a product event ledger.

Every provider callback is normalized before the UI sees it. That is what lets Ripple render one agent experience even when Codex and Claude stream different native shapes.

queuedRun row exists and user projection is recorded.
preparingActive status reserved for setup paths.
runningAdapter is executing and heartbeat updates.
awaiting_approvalProvider is paused on a pending app decision.
cancellingCancellation requested and provider stop is in progress.
completedAssistant projection is saved and status is final.
failedError event and message are persisted.
recoverableStartup recovery found a stale active run.

Shared event types

statusQueued, running, awaiting approval, completed, failed, cancelled, recoverable.
assistantText deltas, final assistant messages, reasoning, and summaries.
toolsTool start, streaming input updates, progress, output, errors, and compacting.
approvalProvider requests are mirrored into pending approval records and updated after response.

Projection path

emitProvider adapter calls sink.emit; runtime inserts a sequenced run event.
fan outThe event bus notifies run-specific and all-run subscribers.
projectAt completion, events become assistant transcript parts and metadata.
persistRun row, provider ids, approvals, and transcript records remain queryable after the process ends.
