Ripple spec · Codex + Claude inside the app

How the agent sees the world inside Ripple

Ripple gives Codex and Claude a product-shaped runtime world: the latest user request, visible chat history, live preview context, project notes, attachments, app policy, and provider-native tools. The model does not continuously watch the app. It receives text and images that Ripple explicitly attaches, then asks for more through tools.

Subsystem: agent runtime Providers: Codex App Server + Claude Agent SDK Visuals: on-demand tools + comment attachments Storage: SQLite run/event transcript
01 · Mental model

One run is a prepared world, not a raw chat message

The app does three jobs before the provider starts: it resolves the workspace, assembles product context, and declares the tool boundary. After that, Codex or Claude uses its own native runtime, while Ripple records every status, message, tool call, approval, and file change back into the same product event stream.

input

User request

Text, pasted file content, uploaded images/files, selected model, mode, provider, and current preview context.

runtime

Ripple context

Project, composition, Main vs revision workspace, preview frame/time, comment anchor, export, and prior visible history.

policy

Instruction stack

App-owned motion-editor policy plus short editable project notes from AGENTS.md or CLAUDE.md.

reach

Tool world

Provider-native file/shell tools, app-managed ripple and hyperframes commands, MCP, and visual tools.

03 · Context stack

What gets assembled before the model answers

The final provider input is layered. Some layers are instruction-channel policy, some are ordinary prompt text, and some are native image/file inputs. This distinction matters: app policy tells the model how to behave, runtime context tells it where it is, and tools tell it what it can inspect next.

Prompt-visible layers latest turn payload
User prompt
The cleaned latest request, plus safe prompt mention instructions for selected tools, skills, agents, files, or folders.
Visible history
Recent product-facing chat messages. Codex always gets this because it starts clean provider threads; Claude gets it when there is no resumable provider session.
Live context
Project, composition, editing target, preview source, frame/time, comment anchor, revision, and export label.
Attachment suffix
A short list of saved attachment paths under .ripple/tmp/agent-attachments/<run>.
Comment visual note
For comment runs only, text pointing to the attached current-frame screenshot or range frame sheet.
Instruction and capability layers provider setup
Ripple policy
Motion-editor behavior: use project/composition language, respect workspace boundaries, prefer HyperFrames skills, keep final replies product-facing.
Visual tool policy
Use native snapshot/frame-sheet tools first when visual context is needed. Normal chats do not get automatic start images.
Project notes
AGENTS.md for Codex or CLAUDE.md for Claude. These are durable team notes, not the app's full system prompt.
Skills
App-managed HyperFrames skills, the Ripple visual context skill, user/project skills, and enabled plugin skills.
Tool env
PATH includes app-managed ripple and hyperframes. Visual context endpoint and bridge tokens are scoped to the run.
04 · Run scenarios

Use the tabs to compare what changes per run type

The runtime path is shared, but the world differs by entry point. A regular chat starts with no automatic visual image. A frame comment or range comment can attach a still or sheet before the provider starts. A revision runs in an isolated workspace.

05 · Provider split

Same product contract, different provider-native channels

Ripple deliberately uses each provider's native integration instead of pretending they are the same API. The shared contract is the runtime event stream, workspace boundary, app policy, visual tools, and approval bridge.

Codex App Server
thread
Starts a clean ephemeral thread with sessionStartSource: "clear". Ripple owns continuity in SQLite.
policy
Project-note fallback goes to baseInstructions; app policy goes to developerInstructions.
sandbox
workspace-write, writable root is the active workspace, network is disabled by default.
visual tools
Dynamic tools ripple.snapshot and ripple.frame_sheet return native image content items.
MCP
Resolved from the local Codex MCP profile for the project lookup path; session init reports connected servers and tools.
Claude Agent SDK
thread
Uses Claude session ids when available via SDK resume; otherwise Ripple adds visible history.
policy
Uses Claude Code preset system prompt with Ripple policy appended through systemPrompt.append.
permissions
permissionMode: "default" for agent mode and "plan" for plan mode, with canUseTool approval routing.
visual tools
A local SDK MCP server named ripple_visual_context exposes ripple_snapshot and ripple_frame_sheet.
skills
Loads app-managed Claude plugins, HyperFrames plugin, visual context plugin, user/project skills, and enabled plugin skills.
06 · What it sees

The model sees explicit context, not the whole desktop

This is the main intuition to keep: a provider run begins from a prepared payload. The agent can inspect more through tools, but it does not automatically know every file, every preview pixel, or every hidden app state.

Visible immediately in prompt/input
Latest user request and safe mention instructions.
Recent visible conversation history, when needed for continuity.
Live context: project, composition, frame/time, preview source, comment, revision.
User attachments, plus automatic comment screenshot or sheet when available.
Reachable by tools on demand
Project files inside the active workspace.
HyperFrames structure, lint, source, and export helpers through bundled commands and skills.
Current preview frame and frame sheets through native Ripple visual tools.
Configured MCP servers that are connected and allowed for the provider runtime.
Not automatic must be supplied
No continuous screen watching or hidden desktop awareness.
No normal-chat start screenshot unless the user attaches one or the agent calls a visual tool.
No network access by default for Codex local runs.
No durable mutation of AGENTS.md, CLAUDE.md, or provider-global settings unless asked.
07 · Tools

The tool surface is app-managed plus provider-native

Ripple's app-managed tool contract is intentionally narrow: make local HyperFrames work possible, make visual context first-class, and keep project boundaries obvious. Provider tools still exist, but approvals and sandboxing keep risky paths visible.

Visual context first-class

snapshot captures the current app frame or an exact timestamp. frame_sheet samples motion over a range. Both return images directly to the model.

Motion project tools local

ripple and hyperframes are placed on PATH. HyperFrames telemetry, update checks, and auto-installs are disabled in the run environment.

Skills and MCP provider-native

App-managed HyperFrames skills, Ripple visual context skill, project/user skills, enabled plugin skills, and configured MCP servers are loaded through each provider's native mechanism.

08 · Approvals

Low-friction local work, visible gates for risky actions

Routine project-local visual checks are allowed quickly. Requests for broader access, network, external paths, tool questions, or file changes are bridged into Ripple's shared approval model so the chat can show an approval card instead of hanging in provider land.

Codex approval bridge

Auto allow: simple ripple snapshot and ripple frame-sheet commands inside the active workspace.

Ask: command execution, file-change approval, permissions, network access, or provider user-input requests.

Warn or deny: paths outside the workspace, unsupported permission references, shell control operators in auto-allow candidates, or approval decisions the app cannot safely send.

Claude approval bridge

Auto allow: native visual MCP tools and simple Bash(ripple snapshot) / Bash(ripple frame-sheet) commands.

Ask: normal Claude tool permission decisions through canUseTool, including edits, commands, network, tools, and questions.

Elicit: MCP form or URL prompts are routed through onElicitation so input requests become product-visible approvals.

09 · Persistence

SQLite is the product memory and event log

Provider threads are not the user-facing source of truth. Ripple persists a normalized run model so chats, comment revisions, approvals, transcript continuity, and UI projection can survive provider differences.

projects root

Local Ripple project path, active composition, project metadata, and archive state.

workspaces where

Main project, isolated chat worktree, or generated-change workspace with safety state.

agent_threads lane

Provider, workspace, conversation/revision target, provider thread/session ids.

agent_runs turn

Prompt, provider, model, mode, status, runtime context JSON, request id, target ids.

agent_run_events stream

Status, activity, assistant deltas, messages, reasoning, tool calls, approvals, usage, errors.

messages UI

transcript_messages preserve runtime continuity; conversation_messages project it into visible chat history.

The important invariant

Provider output becomes product state only after Ripple records it as run events and projects it into messages. That is why Codex can start clean threads while the app still preserves visible continuity.

11 · Verification contract

How to prove the world model still works

A change to this area should prove both the shared product contract and provider-specific behavior. Unit tests catch context assembly. Provider smoke or live evals catch whether Codex and Claude actually receive and use the tools.

Claim Useful coverage Risk caught
Runtime prompt includes live context runtime-context.test.ts, agent-run-context-resolver.test.ts Model edits wrong composition, frame, source, revision, or export target.
Comment visuals attach without blocking comment-visuals.test.ts, comments E2E, live visual-context evals Comment agents lack the frame/sheet they need or wait too long before starting.
Codex receives clean thread plus app policy codex-app-server-adapter.test.ts Provider history leaks, app policy is missing, skills fail, or dynamic visual tools disappear.
Claude uses SDK-native permissions and skills claude-runtime-capabilities.test.ts, claude-agent-sdk-approval.test.ts Claude auto-declines input prompts, bypasses approvals, or loses app-managed plugins.
