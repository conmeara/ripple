Spec - Visual Context

Visual Context

Visual Context is how Ripple lets an agent see an HTML motion world without making the user manage screenshots. Ripple renders HyperFrames compositions, captures the right visual evidence on the user's behalf, and hands that evidence to Codex or Claude so the agent can judge what it is doing visually.

Audience: Motion designers directing agent edits
Surfaces: HTML preview, comments, chats, timeline
Status: Building

User model

The user should never think about screenshots, CLIs, browser sessions, capture backends, endpoints, or generated files. The visible promise is simple: the agent can see the right frame or range, and the app stays fast while it prepares that context.

Model
The agent sees through rendered HTML. Ripple projects are HTML motion compositions. The agent can inspect source files, but visual judgment comes from rendered frames, sheets, and preview-state captures prepared by the app.

Agent Vision Model

The user should be able to ask for a visual change without gathering evidence. Ripple provides the agent with visual context as part of the app, using the current HTML preview, app-managed capture tools, and provider-specific image handoff formats.

Capture journey

The comment or chat environment starts first. Visual context is delivered alongside it by the Visual Context Service, so a frame capture, snapshot, or frame sheet can arrive without making the user wait on infrastructure.

01 - User action
Send from the preview
A comment is anchored to the visible frame or selected range. A chat starts from the current project and preview context.
02 - Environment starts
UI and run begin
The comment card or chat run starts immediately. The user sees their note or conversation state before visual capture finishes.
03 - Visual delivery
Service provides sight
The Visual Context Service captures the needed frame, snapshot, or sheet in parallel and stores it as local runtime evidence.
04 - Agent run
Agent uses the visual
The runtime materializes the local image attachment or exposes app-managed visual tools, then passes concise frame or sheet context to Codex or Claude.
Comment
This mirrors Comments.html. A comment card appears before screenshots, frame sheets, or any agent run. Visual context attaches in the background so the UI is not blocked.

Comment frame

A comment frame is the single still automatically attached to a point comment. It represents the timestamp the user is commenting on.

The generated image is stored for the runtime today, not rendered as a visible thumbnail in the comment card. The user should understand this as "the frame for my comment," not as a generic screenshot.

Range frame sheet

A range sheet is visual evidence for time-based feedback. It samples a selected span so the agent can compare the beginning, middle, and end of a motion note without watching a full render.

Comments create this automatically for selected ranges. Chat tools can also create frame sheets for arbitrary ranges and sample counts within app limits.

Snapshot artifact

A snapshot is an explicit one-off capture requested by chat tools, QA flows, or before/after checks. It can target the current preview state or a specific timestamp.

Unlike a comment frame, a snapshot is not automatically tied to a review note. It is a local artifact the agent can ask for when it needs to inspect a rendered HTML state.

Current preview state

Current preview is not an image by itself. It is the live app state: project, source revision, composition, time, frame, and preview identity.

Chat visual tools use this state when the agent asks for ripple snapshot --at current. The service then turns that state into a snapshot image or frame sheet.

States

The capture state should never replace the user's note state. In the implemented comments path, it is a quiet attachment lifecycle underneath the card. In chat, visual context is mostly runtime context plus on-demand ripple snapshot and ripple frame-sheet tools.

visible
Note exists
The comment card is already visible and editable in the user's flow before capture starts.
background
Comment capture queued
Ripple records the project, composition, preview source, time, source revision, and range request.
service
Coordinator running
The Visual Context Service queues by session key, allows two active captures, and falls back between registered backends.
attachment
Context materialized
The agent run receives local image inputs plus concise prompt context; user attachments take precedence when budgets are tight.
fallback
Text still works
If capture fails, the user keeps the note and the agent still receives text context. Runtime visual tool failures are reported as command errors, not app data loss.

UI treatment

  • Current UI does not show generated visual thumbnails. Comment cards show timecode, status, actions, and user-supplied attachment chips; generated screenshotPath is stored for runtime use.
  • A thumbnail or frame-sheet chip is a proposed enhancement. It can reassure the user that the agent can see the relevant visual, but the current implementation keeps that evidence behind the scenes.
  • Capture failures should be quiet by default. Use product language such as "Could not attach visual context" only when the user needs to act.
  • Debug terms stay out of normal UX. Do not show backend names, ports, endpoints, handoff files, workspaces, or provider plumbing in comments and chats.

Capture Strategy

The product goal is not one universal capture path. The goal is to choose the fastest method that gives the agent enough visual quality for the route: instant comment context, reliable chat self-checks, and accurate frame sheets when timing matters.

Strategy Best for Strength Risk / test focus
preview-surface capturePoint comments at the frame already visible in the preview player.Likely fastest route because the user is already looking at the frame.Must prove the captured pixels match the painted preview, active revision, and playhead before using it as the hot path.
engine / producerHigh-quality chat snapshots, deterministic QA frames, and correctness fallbacks.Re-renders the HTML composition through app-owned capture backends.Can be slower; tests should measure latency and visual fidelity against the preview surface.
HyperFrames CLIAgent-requested snapshots and frame sheets, packaged tool paths, and standalone fallback.Works as an explicit tool contract for Codex and Claude.Must stay available as bare ripple inside provider shells and should not require extra setup by the user.
cache / reuseRepeated captures of stable frames or ranges while the user reviews the same composition.Can remove repeated render work and make chat checks feel immediate.Invalidation is hard: edits, revision switches, composition changes, media changes, and timeline changes must retire stale visuals.
low-res profileFast visual grounding when the agent only needs layout, presence, color, or timing landmarks.Smaller images move faster through provider attachment limits and reduce capture cost.Tests must define where low-res is enough and where text, small logos, or fine motion require higher quality.
Decision
Route selection needs tests, not taste. Chat current-frame checks, chat arbitrary-frame checks, point comments, and range comments should each have latency and visual-quality fixtures so Ripple can pick preview capture, service re-render, CLI capture, cache, or low-res capture intentionally.

Local artifacts

Generated visual context artifacts are local and scoped. They are runtime evidence, not ordinary source changes, and they should not pollute the project unless the user intentionally imports them as assets.

Comment visuals

Frame captures for comment runs are copied into canonical project storage so the visual stays available even if a proposed revision is later deleted.

.ripple/comment-visuals/<thread-id>/frame.png

Comment frame sheets

Range comments store a sheet plus manifest so the agent can inspect sampled timecodes and frames without a full render.

.ripple/comment-visuals/<thread-id>/sheet.png

CLI frame sheets

Explicit ripple frame-sheet calls write a sheet, manifest, and copied frame images for review or provider use.

.ripple/frame-sheets/fs_<id>/sheet.png

Snapshots

Explicit ripple snapshot calls live under the project-local Ripple directory and are guarded against traversal or symlink escape.

.ripple/visual-context/snapshots/snap_<id>/

Provider attachments

Stored visuals are materialized into run-scoped temp files before being sent as Codex local images or Claude native image/PDF content.

.ripple/tmp/agent-attachments/<run-id>/

Endpoint and bridge

Agent tools communicate through token-auth localhost endpoints or polling request/response files when an endpoint is unavailable.

.ripple/agent-visual-context/<run-id>/requests/

Optional eager handoff

Eager run-start visual manifests are only prepared when RIPPLE_EAGER_AGENT_VISUAL_CONTEXT=1; that is not the default chat path.

.ripple/agent-visual-context/<run-id>/manifest.json

Edge cases

Capture fails

The comment remains and the agent still receives text context. Chat visual tool failures should be handled as command output, not as lost app state.

Source changes mid-capture

Generated context should be invalidated or regenerated rather than reused as if it still matches the preview.

Range is reversed or tiny

Normalize the selection and sample within bounds. A single-frame range can fall back to current-frame behavior.

Agent asks for more visuals

Use app-managed ripple snapshot and ripple frame-sheet commands, not ad hoc browser capture or package installs.

Endpoint is unavailable

Provider runs can fall back to the file bridge. If both startup paths fail, cleanup runs and the agent continues without app-managed visual tools.

Attachment budget is full

Drop automatic visual context before dropping user-provided attachments, and keep the user-facing reply concise.

New chat first message

The embedded new-chat form currently does not receive live preview runtime context, so do not promise automatic first-message visual capture until that path is wired.

Current frame mismatch

--at current requires verified app-managed visual context. Missing or mismatched runtime context should fail clearly instead of silently guessing a frame.

Local image viewing is unavailable

The agent should report the sheet path and manifest details so Ripple can show the artifact instead of repeatedly trying external viewers.

Verification posture

  • Spec changes stay behavior-first. Test names should read like product claims, not implementation trivia.
  • Performance stays visible. Keep timing expectations in the quality matrix because visual context is only useful when comments stay fast and active chats can request visuals without setup detours.
  • Provider proof matters for final confidence. The live visual-context E2E is the proof that app-owned capture reaches a real agent path.
