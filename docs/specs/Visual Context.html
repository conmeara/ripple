Spec - Visual Context - v2

Visual Context

Agents see the user's projects through code. They can read files, but it is often hard for them to understand exactly how the work renders. Visual Context gives agents a fast way to see the frame or range they are working on so they can make better changes.

Ripple provides visuals as a service to agents. The goal is speed and quality: the user keeps working, and the agent gets enough visual evidence to understand the project.

Audience: Motion designers directing agent edits
Surfaces: Preview, comments, chat
Status: Refactor target - ideal system

Architecture

We ran the render-path benchmark across every available capture method. The result was clear: live preview for the current frame, warm Engine for certain frames and frame sheets. The app is coded to these routes - the spec picks once, and the benchmark stays as a regression check.

Decision
Live preview for the current frame. Warm Engine for everything else.

Current-frame capture is fast enough off the live preview surface to feel immediate. Arbitrary timestamps and frame sheets need source-canonical rendering, not display pixels.

Coded routes
  • CurrentLive preview surface capture, with normalization guards.
  • CertainWarm Engine render at the requested timestamp.
  • SheetWarm Engine sampled renders assembled into a compact frame sheet.

Agent interface

Agents never pick backends or set up browsers. They call two clean commands; the app dispatches each to its coded route.

Case Agent command Coded route Reason
Current app view ripple snapshot --at current Live preview capture Fastest path. Reflects exactly what the user sees.
Specific timestamp ripple snapshot --at 1.25s Warm Engine render Source-canonical. Does not depend on visible preview state.
Motion over time ripple frame-sheet --range 0s..8s --samples 8 Warm Engine renders + sheet assembly Best tradeoff between motion reasoning and payload size.

What we learned

  • Warm sessions matter. Once the renderer is alive, multi-frame capture becomes plausible for agent use.
  • Fidelity is clean on the simple fixture. Painted preview and fast-browser output matched pixel-for-pixel at SSIM 1.000 in the test fixture.
  • Live preview is the right hot path for current frame. 9 ms once ready, with stable geometry across compact, wide, and tall layouts. Pixel deltas vary by layout, which is why the route is "what the user sees," not source-canonical comparison.

Rules

Invariants the implementation should preserve across comments, chats, visual QA, and provider runs.

Ripple owns visual capture

Agents use Ripple visual commands. They do not set up browser capture or choose capture backends.

Product language stays simple

Current frame, selected range, snapshot, frame sheet, preview. Backend names stay out of UX.

Source-aware capture

Visual context respects whether the user is viewing Main or a proposed revision.

Local and project-scoped

Visual artifacts are local. They do not appear as normal source changes or assets unless the user imports them.

Sheets stay compact

Enough samples to understand motion, not a huge payload. Sampling and column bounds are bounded.

User attachments first

User-provided attachments take precedence over automatic visuals when attachment limits are reached.

Stale visuals invalidate

Source changes invalidate stale visuals. If Ripple is not sure the visual still matches, it makes a fresh one.

One artifact, then adapt

Ripple prepares one local artifact, then sends it to the chosen agent provider in the right format.

Vision

The user leaves feedback without waiting. The agent can see the frame or range the user meant. Ripple captures a local visual artifact, attaches it to the agent run in the provider's native format, and keeps the image path plus manifest available for timing details and fresh visual checks.

Testing posture

  • Test state, not interactions. Assert on the captured artifact, the agent message that goes out, the file on disk - not on which functions the dispatcher called. Tests that verify call sequences break on refactor even when behavior is unchanged.
  • DAMP over DRY. Each test reads like a behavior spec on its own. Duplicated arrange code is fine when it keeps a test independently understandable.
  • Real implementations over mocks. Prefer real warm sessions, real PNG outputs, and real fixtures. Mock only at provider boundaries where real deps are slow or non-deterministic.
  • Pyramid shape. Most tests are unit (capture logic, dispatcher, sampling, invalidation). Integration covers seams (CLI delegation, comment thread + visual write). E2E is reserved for full user flows (comment-to-agent-message, live provider).
  • Prove-it for refactor bugs. Any regression caught during the refactor toward this spec gets a reproduction test that fails before the fix.
  • One assertion per concept. Latency and fidelity stay in separate tests, even though they share a fixture. Comment-waits-for-visual and chat-does-not-wait stay in separate tests.
