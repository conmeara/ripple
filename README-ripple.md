Ripple is a local-first motion-graphics app built on HyperFrames, and the core architecture is that HyperFrames is the motion engine while Ripple is the product workflow around it: HyperFrames should own composition loading, runtime timing, nested compositions, clip manifests, player behavior, snapshots, renders, and future framework-level motion semantics, while Ripple should own the concise app layout, project-first onboarding, composition and asset browsing, preview chrome, embedded timeline UI, chat, frame-anchored comments, reviewable revisions, export workflows, user-facing language, loading/error states, and source-write safety rules. The best pattern is a Ripple-owned adapter around official HyperFrames primitives: use `@hyperframes/player` for playback, ask the Electron main process for approved project and preview data, serve files through validated project-boundary routes such as `ripple-preview:`, normalize HyperFrames runtime data into Ripple-owned shared models, and render the visible UI with Ripple components instead of embedding the full HyperFrames Studio shell. Studio is valuable reference material for interaction patterns and information architecture, but the normal Ripple product surface should stay native, focused, and app-like. Renderer code should not scan arbitrary absolute paths, launch privileged commands, or decide whether a file belongs to a project; project, composition, asset, preview, timeline, render, export, and future editing operations should resolve through typed main-process APIs using project IDs. Any change to the HyperFrames wrapper boundary should keep the HyperFrames package family pinned together and run `bun run test:hyperframes` so adapter drift, package-surface drift, runtime-message drift, and unsafe Studio imports are caught early. The decision rule for future contributors is simple: if it is motion semantics, prefer HyperFrames; if it is product workflow, Ripple owns it; if it touches files, commands, rendering, exports, or source writes, route it through the main process; if Studio does something useful, borrow the concept and wrap it in Ripple's model and UI; and if a choice would make users think about repos, paths, dependencies, or dev servers, hide it behind motion-design language unless it is an advanced or debug surface.
